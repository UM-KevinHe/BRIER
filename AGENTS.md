# Part 1: For contributors

## What BRIER is, briefly

BRIER (Bregman-divergence-based regularized integrative estimator in regression) is an R package for transfer-learning genetic risk prediction. Target-cohort prediction is improved by borrowing information from external cohorts, pretrained external coefficients, or summary-level association statistics, under high-dimensional regularization. Three top-level fitting functions cover the three external-data regimes: `BRIERfull` (raw external), `BRIERi` (pretrained coefficients with individual-level target), `BRIERs` (pretrained coefficients with summary-level target).

The package is roughly 78% R and 21% C++. Heavy numerical work (coordinate descent solvers, LD assembly) lives in C++ via Rcpp + RcppArmadillo; high-level orchestration, model selection, and plotting live in R.

License: GPL-3. Maintainer: Ruiwen Zhou (zrayw@umich.edu).

## Standard build sequence

After any change to R sources, roxygen blocks, or C++ source, run:

```r
Rcpp::compileAttributes()    # if any src/*.cpp signature changed
devtools::document()         # regenerate man/*.Rd + NAMESPACE from roxygen
devtools::check()            # full R CMD check
```

For pkgdown site rebuilds (vignettes, llms.txt, reference pages):

```r
pkgdown::build_site()
```

This emits to `docs/`. The site is deployed via GitHub Pages from the `docs/` directory on the main branch. Note: `pkgdown::build_site()` copies `pkgdown/assets/llms.txt` to `docs/llms.txt` as part of the build, so updating `llms.txt` requires:

1. Edit `llms.txt` at repo root (the canonical copy).
2. Mirror to `pkgdown/assets/llms.txt` (or set up a symlink in `pkgdown/_pkgdown.yml`).
3. `pkgdown::build_site()` to push to `docs/llms.txt`.
4. Commit and push.

## Function naming conventions

The package has mixed naming because the user-facing surface has both R-style and biostat-tradition names. Do not rename existing functions to enforce a single convention. When adding new functions, match the nearest neighbor in the same R file:

- Top-level fitters: `BRIERfull`, `BRIERi`, `BRIERs` (camelCase suffix, no separator)
- Selection / CV companions: `BRIERfull.selection`, `BRIERi.cv` (S3-style dotted suffix, but NOT actual S3 methods)
- Preprocessing: `preprocessI`, `preprocessS`, `mergeExternals` (lowerCamelCase)
- Utilities: `calLD`, `p2cor`, `evalMetric`, `getLDB` (lowerCamelCase with abbreviations)
- Plotting: `plot.box`, `plot.eta`, `plot.importance` (dotted names, but these are NOT S3 methods despite looking like them; call them directly, not via `plot()`)
- LD internals: `LD_blocks`, `LD_sigma` (snake_case)

The dotted-name-not-S3 pattern is load-bearing: these functions are exported under their dotted names and have their own argument signatures incompatible with `?plot`'s generic. Do not register them as S3 methods.

## C++ source discipline

When editing `src/*.cpp`:

1. Edit the `.cpp` file, not `RcppExports.cpp`.
2. Run `Rcpp::compileAttributes()` from the package root. This regenerates `src/RcppExports.cpp` and `R/RcppExports.R`.
3. Run `devtools::document()` so any roxygen `@useDynLib` or `@importFrom` lines update.
4. Run `devtools::check()` to catch ABI mismatches early.

Common footgun: if you change a C++ function signature but forget step 2, R-side calls compile against the stale `RcppExports.R` and crash at runtime with cryptic Armadillo errors. The error message rarely points at the signature change.

## roxygen2 documentation style

Every exported function has a roxygen block. Conventions:

- `@param` lines should be terse but specific. State the expected shape (matrix dimensions, vector length, allowed values).
- `@return` describes the structure of the returned object, including the class attribute.
- `@examples` should be runnable. The vignette examples are verified to run cleanly; the `?function` Rd examples have not been thoroughly tested and may have drift. When editing existing examples, prefer porting a snippet from the vignettes over rewriting from scratch.
- `@seealso` is used for cross-references between fitters and their selection / cv companions. Keep these bidirectional.
- `@references` should match the format used in existing files (look at `R/BRIERfull.R` or similar). Use `\url{}` for URLs and `\emph{}` for journal names.

## Style: no em-dashes in prose

User-facing prose in `README.md`, `AGENTS.md`, `llms.txt`, vignettes, and roxygen `@description` blocks should use commas, colons, or parentheses instead of em-dashes (`—`). Code comments, function-index entries, and bullet lists where dashes are structural markers are fine. This is a deliberate style choice; the rule is "long-form prose, no em-dashes."

## Avoid pinning version numbers in user-facing prose

`llms.txt`, `README.md`, vignette text, and roxygen `@description` blocks should describe what the package currently does, not what version it does it in. Version-specific language ("as of 1.0.2", "in version 1.0.1", "supported since 1.0") becomes stale fast and is tedious to keep in sync across files. The package metadata (`DESCRIPTION`) is the single source of truth for the current version.

Acceptable: "currently supported families: gaussian, binomial, poisson."
Avoid: "in 1.0.2, supported families are gaussian, binomial, poisson."

The exception is `NEWS.md` and other changelog-shaped files, where version pinning is the entire point.

## Silent-failure traps to remember when editing

These are the same ones documented in `llms.txt` for users, but contributors editing the relevant functions should also be aware:

- `BRIERi()` requires `beta.external` with an intercept slot as the first row. `BRIERs()` does NOT. Do not unify these unless intentionally and with a deprecation cycle.
- `BRIERs()` returns standardized coefficients. `BRIERs.selection()` does not standardize internally. If you add validation-set tuning to `BRIERs.selection()`, do not silently standardize the inputs; require the caller to standardize. The asymmetry with `BRIERfull.selection()` / `BRIERi.selection()` (which leave it to the caller's training preprocessing choice) is intentional.
- `BRIERfull()` cannot fit a target-only model. `eta.list = 0` alone is not a supported configuration; the function relies on having at least one positive borrowing weight in the grid. Users who want a target-only baseline should use `BRIERi()` with `eta = 0`. If you change this behavior (e.g., to allow `BRIERfull(eta.list = 0)` to return a sensible target-only fit), update `llms.txt`'s "BRIERfull cannot fit a target-only model" pitfall AND the eta-range diagnostic section in the same PR.
- `family = "cox"` is on the roadmap for `BRIERfull()` and `BRIERi()` but will never be supported for `BRIERs()` (summary-level workflow is incompatible with Cox partial likelihood). Do not accept `"cox"` as a family argument in `BRIERs()`.
- `family = "negbin"` is documented in `Appendix_Distr` but not yet functional for any fitting function. If you wire it up, also update `llms.txt` and `Appendix_Distr` together.
- `BRIERi.cv()` does NOT standardize internally. If a user pre-standardizes `X` or pre-residualizes `y` on the full sample, CV estimates leak across folds. This is documented but easy to miss when reviewing PRs that change `BRIERi.cv()` semantics. Resist the urge to add internal standardization "for safety"; it would silently break what users who DID pre-standardize already expect.
- `calLD()` drops constant columns and returns their retained indices in `$nz`. `SNP.info` and `sumstats` must be subset by `$nz` to stay aligned. This is the caller's responsibility, not `calLD()`'s.
- `getLDB()` returns BED files with `"chr1"`-style chromosome labels, but `SNP.info$CHR` is typically numeric. The user must convert with `as.numeric(sub("chr", "", LDB[, 1]))` before passing to `calLD()`. If you wrap `calLD()` in a higher-level helper, consider doing this conversion inside so the wrapper "just works."
- `multi.method = "ind"` is multiplicative in M and impractical for M >= 5. If you add new aggregation methods, document the scaling behavior explicitly in the roxygen `@param`.
- `penalty.factor` follows the glmnet convention (length-p numeric, `0` = unpenalized) across all three fitters. Keep this contract: do not introduce a fitter where `penalty.factor`'s semantics differ.

# Part 2: For end users (writing analysis code)

## First thing: read `llms.txt`

[`llms.txt`](./llms.txt) is the authoritative user-facing reference. It gives you the decision tree, parameter semantics, worked examples drawn from verified vignettes, and the full list of common pitfalls in one compact file. If you are using an AI assistant (Claude, ChatGPT, Gemini, Cursor, or similar) and you have not yet pointed it at `llms.txt`, do that first. Either share the URL `https://um-kevinhe.github.io/BRIER/llms.txt`, or paste the file contents into your conversation.

The rest of this section is a high-level orientation, not a substitute for `llms.txt`.

## Which function do I use?

The shortest version of the decision tree:

- Target data is **summary statistics only** (GWAS sumstats + LD): use `BRIERs()`.
- Target data is **individual-level**, external is **pretrained coefficients**: use `BRIERi()`.
- Target and external are both **raw individual-level data**, and sample sizes are roughly comparable: use `BRIERfull()`.
- Target and external are both raw, but external is much larger (e.g. 40,000 vs 2,000): `BRIERfull()` may not gain much over `BRIERi()` applied to a model fit on the external. Case by case.

For multiple external models, also use `mergeExternals()` to combine them, set `multi.method = "stacking"` (recommended default; avoid `"ind"` for M >= 5 due to compute cost), and consider running a per-external diagnostic pass first before joint integration.

## The six things to remember (silent-failure traps)

These are the patterns that look like working code but produce wrong answers:

1. **Always specify `family` explicitly**, especially for `BRIERs()`. A binary outcome with `family = "gaussian"` will silently return linear predictors from `predict(..., type = "response")` instead of probabilities. There is no error at fit time.

2. **`BRIERi()` requires an intercept slot** as the first row of `beta.external`. If your external coefficients do not include an intercept, prepend a row of `0`. `BRIERs()` does NOT require this.

3. **`BRIERs()` returns standardized coefficients**, and `BRIERs.selection()` does not standardize its inputs. Validation `X` must be standardized (e.g. via `standardize_X()`) before passing to `BRIERs.selection()`. Standardize `y` only for `family = "gaussian"`; pass `y` raw for binomial or poisson.

4. **`BRIERi.cv()` does not standardize internally.** If you pre-standardize `X` or pre-residualize `y` on the full sample before calling `BRIERi.cv()`, CV estimates leak across folds and become optimistic. Either pass raw inputs, or standardize inside each fold yourself.

5. **`calLD()` drops constant columns** and reports retained indices in the return `$nz` slot. Subset both `SNP.info` and `sumstats` by `$nz` after calling `calLD()` to keep alignment. Forgetting to subset one of them silently breaks downstream alignment.

6. **`BRIERfull()` cannot fit a target-only model.** If you want a no-borrowing baseline for diagnostics or comparison, use `BRIERi(eta = 0)` instead, even if raw external data are available. `BRIERfull(eta.list = 0)` is not supported and will not return a sensible target-only fit.

## Two things that are NOT currently supported

- `family = "cox"` for any fitter (planned for `BRIERfull` and `BRIERi`; never for `BRIERs`).
- `family = "negbin"` for any fitter (planned; not yet functional).

If `Appendix_Distr` advertises either of these, it is documenting the roadmap. Do not use them in your scripts; check the package changelog (or test against the installed version) before relying on either.
